# Configuration

Settings are layered: every infra knob lives on `CoreSettings`
(`src/core/settings.py`) and application-specific fields live on
`Settings` (`src/common/settings.py`). The runtime profile is selected
by the `APP_ENV` env var.

## Precedence

Highest → lowest:

1. **AWS Secrets Manager** — when `aws_secret_name` is set, every key
   in the JSON blob overrides everything below.
2. **Environment variables** — case-insensitive; lists/dicts use JSON
   (`AUTH_ENABLED_PROVIDERS='["jwt","api_key"]'`).
3. **`.env`** at the project root.
4. **Field defaults** declared on the class.

The merge happens in `CoreSettings.settings_customise_sources`;
secrets manager failure logs + degrades but never raises (so a dev
process boots even without AWS).

## Profile classes

`APP_ENV` resolves to a subclass of `Settings`:

| `APP_ENV` | Profile | Behaviour |
|---|---|---|
| `local` (default) | `LocalSettings` | Permissive, `DEBUG` logging. |
| `test` | `TestSettings` | `db_name=app_test`, noop audit, metrics off. |
| `dev` | `DevSettings` | Fails fast if `DB_HOST` or `FIELD_ENCRYPTION_KEY` missing. |
| `uat` | `UatSettings` | Inherits dev validator; INFO logging. |
| `prod` | `ProdSettings` | Requires `FIELD_ENCRYPTION_KEY`, `SECRET_KEY`, non-localhost `DB_HOST`. |

Unknown `APP_ENV` values fall back to `LocalSettings` with a warning.

To add a profile-specific override, **override the default on the
subclass** — do not duplicate the field on `Settings`. Cross-field
invariants belong in `@model_validator(mode="after")` so the app
fails fast on boot rather than on first request.

## Reading settings inside core

`src.core.*` modules read configuration through
`core.runtime.get_settings()`. This indirection is the layering rule:
`src.core` must never import `src.common`. The application's
`lifespan` calls `configure(settings)` once at startup so `get_settings()`
sees the resolved profile instance.

## Settings catalog

The full env-var matrix lives in
[`environment.md`](environment.md), regenerated by
`scripts/dump_settings_schema.py`. Run `python scripts/dump_settings_schema.py --check`
to verify the doc matches the schema; `--write` regenerates it.

## Verifying a profile

```bash
APP_ENV=prod python -c "from src.common.settings import settings; \
    print(type(settings).__name__)"
```

Should print `ProdSettings` *and* fail loudly if the required prod
secrets are missing — the validator catches the gap before the
process accepts traffic.
